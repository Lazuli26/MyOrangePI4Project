# Orange Pi NAS Setup

## Purpose

This note records the NAS configuration prepared for the Orange Pi board on `2026-06-19`.

Goal:
- expose the attached `500 GB` USB drive to the local network over SMB/CIFS
- keep the setup low-risk and easy to repeat from this repo

## Board And Storage Context

- Board IP: `192.168.50.71`
- Active management path: `SSH`
- Root storage remains on `SD card`
- Attached USB drive detected as:
  - device: `sda1`
  - size: about `465.8G`
  - filesystem: `vfat`
  - label: `KUBUNTU 24_`
  - UUID: `1C0D-3B51`

## Chosen NAS Approach

- Service: `Samba`
- Network protocol: `SMB`
- Mount path on the board: `/srv/nas/usb500gb`
- Share name: `usb500gb`
- Authentication model:
  - authenticated share
  - user: `orangepi`
  - Samba password defaults to the same value passed in as `SMB_PASS`
- Network scope:
  - restricted to the local `192.168.50.x` LAN in the share definition

Why this approach:
- Windows, macOS, Linux, and most media clients can access SMB easily
- the attached disk is already `vfat`, so it can be shared without reformatting
- the configuration uses the partition `UUID`, so the mount remains stable across reboots
- `wsdd` can advertise the share to modern Windows systems so it is more likely to appear in File Explorer `Network`

## Mount Configuration

The automation adds this `fstab` entry:

```fstab
UUID=1C0D-3B51 /srv/nas/usb500gb vfat uid=1000,gid=1000,fmask=0002,dmask=0002,utf8=1,shortname=mixed,nofail,x-systemd.automount,x-systemd.idle-timeout=5min 0 0
```

Notes:
- `uid=1000,gid=1000` maps the files to user/group `orangepi`
- `fmask=0002,dmask=0002` gives group write access while avoiding world-write
- `nofail` avoids boot failure if the drive is unplugged
- `x-systemd.automount` mounts on demand instead of forcing early boot work

## Automation

Use:
- `install_nas_remote.sh`

Environment variables:
- `SUDO_PASS`
- `SMB_PASS`

Optional overrides:
- `USB_UUID`
- `MOUNT_DIR`
- `SHARE_NAME`
- `UNIX_USER`
- `UNIX_GROUP`
- `LAN_PREFIX`

Example from the board itself:

```bash
SUDO_PASS='orangepi' SMB_PASS='orangepi' ./install_nas_remote.sh
```

## Expected Client Path

Primary LAN path:

```text
\\192.168.50.71\usb500gb
```

Credentials:
- username: `orangepi`
- password: whatever was supplied as `SMB_PASS`

Network discovery note:
- the board can still be opened directly with `\\192.168.50.71\usb500gb`
- to improve visibility in File Explorer `Network`, install and enable `wsdd` on the board
- even with `wsdd`, Windows discovery can still depend on the client PC using a `Private` network profile with network discovery enabled

## Verification Commands

On the board:

```bash
findmnt /srv/nas/usb500gb
sudo systemctl status smbd nmbd wsdd --no-pager
sudo testparm -s
sudo ss -tlnp | grep -E ':(139|445)\s'
```

From a client:

```text
\\192.168.50.71\usb500gb
```

## Risks And Notes

- The drive is `vfat`, so Unix ownership and permissions are synthetic mount-time mappings rather than native filesystem metadata.
- The board is still Wi-Fi managed, so SMB performance and reliability depend on wireless quality.
- Credentials on this board are still documented as default/simple; rotate them if this share will remain in use.
